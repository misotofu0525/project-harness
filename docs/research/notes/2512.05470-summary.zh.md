## 一句话结论

这篇论文的核心观点是：`context engineering` 不应该继续停留在 prompt、RAG、tool use 的零散技巧层，而应该升级为一种可持久化、可治理、可审计的软件架构；作者提出的统一抽象是把上下文视为“文件系统中的资源”。

## 论文要解决什么问题

现有 GenAI / agent 系统里，上下文通常是碎片化的：

- prompt 是一次性的
- RAG 只解决检索，不解决治理
- tool integration 往往是点状接入
- memory 往往是 ad hoc 的外挂模块
- 缺少统一的 provenance、versioning、audit 和 access control

作者认为，这会带来几个直接问题：

- 上下文生命周期不可追踪
- 多源上下文难以统一管理
- context rot / knowledge drift 难以发现
- 人类输入、工具、记忆、外部知识之间缺少统一治理
- 系统难以验证、复盘和工业化落地

## 核心主张

作者借用 Unix 的 `everything is a file`，提出：

- 把 memory、history、tool、external API、human input 都映射成统一的“文件系统资源”
- 用统一 namespace、metadata、mounting、access control 来管理它们
- 在这个基础上构建一条可验证的 `context engineering pipeline`

换句话说，论文不是在讲“怎么写更好的 prompt”，而是在讲“怎样把上下文当成基础设施来设计”。

## 提出的整体架构

论文架构可以拆成两层。

第一层是 `File System as Infrastructure for Context`：

- 用文件系统抽象统一承载异构上下文资源
- agent 和 human 都像 OS process 一样对这些资源做 `read / write / search / exec`
- 资源通过 mount 接入，后端可以是文件、向量库、知识图谱、MCP、外部 API

第二层是 `Context Engineering Pipeline`：

- `Context Constructor`
- `Context Updater`
- `Context Evaluator`

这条 pipeline 负责把长期存在的上下文资源，变成一次具体推理所需的有限上下文窗口。

## 这套架构依赖的 6 个软件工程原则

- `Abstraction`
  用统一接口隐藏知识图谱、memory store、API、工具等底层差异。
- `Modularity and Encapsulation`
  每个上下文源都是一个边界清晰、可替换的 mounted component。
- `Separation of Concerns`
  明确区分 data、tool、governance，不把所有东西都混成 prompt。
- `Traceability and Verifiability`
  所有读写、变换、调用都应该留下可追踪记录。
- `Composability and Evolvability`
  新 context source 可以增量挂载，不需要重写整个系统。
- `Governance`
  上下文不仅要能用，还要可控、可审计、可回放。

## Persistent Context Repository 是什么

论文把持久上下文仓库拆成 3 类对象：

- `History`
  不可变原始事实源，记录用户输入、agent 输出、中间推理步骤和环境交互。
- `Memory`
  从 history 中提炼出来、适合检索和推理的结构化视图。
- `Scratchpad`
  临时工作区，用于中间草稿、假设、临时计算结果。

作者的意思不是只做一个“记忆库”，而是把上下文生命周期分层：

- 原始事实放 history
- 可复用知识放 memory
- 临时推理痕迹放 scratchpad

这比把所有东西都塞进同一种 memory store 更清楚。

## Context Engineering Pipeline 的 3 个部件

### 1. Context Constructor

负责从持久仓库中：

- 选择相关上下文
- 排序和裁剪
- 压缩
- 生成本次推理的 `context manifest`

这里最重要的思想是：

- 上下文装配过程本身也应该是显式 artefact
- 不只是“拿了哪些内容”，还要记录“为什么拿、为什么没拿”

### 2. Context Updater

负责把已经构造好的上下文装入模型的有限 token window，并在长会话里持续刷新：

- 初次静态注入
- 推理中增量流式补充
- 根据对话和反馈进行 adaptive refresh

它解决的是：上下文不是一次性喂完就结束，而是要和运行时状态同步。

### 3. Context Evaluator

负责在模型产出后：

- 检查 hallucination、contradiction、context drift
- 把验证通过的结果回写 memory
- 对不确定结果触发 human review
- 把评估指标写进 metadata

这一步很关键，因为它把“上下文工程”从单向输入，变成了闭环系统。

## 论文认为必须正视的 3 个设计约束

- `Token window`
  模型上下文窗口有限，所以长期上下文必须被选择、压缩、分层和增量注入。
- `Statelessness`
  模型天然无状态，必须把状态外置到持久上下文仓库。
- `Non-determinism`
  模型输出是概率性的，所以必须保留输入输出对、元数据和 provenance，才能审计和复现。

这三点基本就是作者为什么要从“prompt 技巧”升级到“架构设计”的理由。

## AIGNE 实现说明了什么

论文不是纯概念文。作者在 AIGNE 框架里做了实现，说明这套设计不是空中楼阁。

它做了几件关键事情：

- 用 `AFS` 作为 Agentic File System 接口
- 把 memory、history、MCP、外部模块都挂载到统一命名空间
- 暴露统一的 `list / read / write / search / exec` 操作
- 用 metadata 和 lineage 支持 audit / rollback / replay

## 两个 exemplar 的意义

### Exemplar 1: 持久记忆 agent

重点不是“agent 有记忆”这件事本身，而是：

- memory 被视为文件系统中的资源
- session 之外仍然可重建
- 记忆进入推理前先经过结构化和治理

### Exemplar 2: MCP + GitHub

重点不是“接了 GitHub MCP”，而是：

- MCP server 也能被 mount 成文件系统模块
- agent 对 GitHub 的交互可以被统一成 context resource + executable action
- 外部工具不再是 prompt 里的一段说明，而是被纳入统一上下文基础设施

## 这篇论文最有价值的地方

- 它把 `context engineering` 从经验技巧提升到了架构层
- 它把上下文看成有生命周期的 artefact，而不是一次性 prompt
- 它强调 `manifest / metadata / lineage / audit`，这对工业系统很重要
- 它把 human 直接放进体系里，不是只把人当审批按钮

## 这篇论文的边界和不足

下面几条是基于论文内容的判断，不是作者逐字原话：

- 这更像一篇架构论文，而不是强 benchmark 论文
- 它证明了“这样设计是合理且可实现的”，但没有充分证明“这比其他方案量化更优”
- 文件系统抽象很强，但也可能偏重，落地时未必所有团队都需要完整采用
- 真正难的地方仍然在 metadata 质量、context selection 质量和 evaluator 质量

## 对我们有什么直接启发

如果把它翻译成我们日常用 Codex / agent 的工程动作，大概是这些：

- 不要把关键上下文只藏在 prompt 里，要落成显式 artefact
- 把 `history / memory / scratchpad` 分开，不要混成一个大杂烩
- 每次上下文装配都最好留下 manifest
- context 选择、压缩、刷新、验证应该是独立环节
- 工具、外部知识、规则、人工反馈都应该有统一位置和元数据
- human review 不是补丁，而是架构内的正式节点

## 和上一篇文章的关系

它和前面那篇 “How To Be A World-Class Agentic Engineer” 不是冲突关系，而是视角不同：

- 前一篇更强调：少上下文、少依赖、拆分任务、控制 agent 行为
- 这篇论文更强调：把上下文做成可治理的基础设施

合在一起看，比较合理的结论是：

- `战术上` 要尽量保持单次上下文干净、最小、任务化
- `战略上` 要把长期上下文、记忆、工具和人工反馈沉淀成可追踪系统

## 我的简版判断

如果你只记住一句话，那就是：

`Prompt 只是上下文的投递形式，不是上下文本身；上下文应该像代码和数据一样被建模、存储、治理和审计。`
