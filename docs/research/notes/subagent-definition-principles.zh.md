# Subagent 定义原则

- 来源 1: [openai-codex-best-practices-summary.zh.md](openai-codex-best-practices-summary.zh.md)
- 来源 2: [context-engineering-experience-map.zh.md](context-engineering-experience-map.zh.md)
- 来源 3: `2026-03-22` 本仓库关于 subagent 设计的交互式讨论与综合判断

## 一句话判断

subagent 的核心价值不是“多开几个 agent”，而是做 `context firewall`：把主 agent 不该长期携带的高噪音、高局部性、强技能依赖的工作隔离出去，只把压缩后的结论带回主线程。

## 一、先区分两件事

### 1. 使用一个 subagent

这是任务级决策。

核心问题是：

- 这个子任务是否应该从主线程剥离出去执行？

### 2. 定义一个 custom subagent role

这是系统级决策。

核心问题是：

- 这个模式是否值得长期沉淀成一个可跨任务、跨项目复用的专用角色？

这两件事不能混为一谈。

- 很多任务值得 `spawn subagent`
- 但只有少数模式值得定义成 `custom role`

## 二、为什么 subagent 对上下文管理有价值

主 agent 最应该保留的是：

- 目标
- 约束
- 关键决策
- 最终结论

而很多任务过程本身是高噪音的，例如：

- 多轮代码搜索
- 错误路径较多的网页搜索
- 日志翻查
- 文档检索
- 失败的复现尝试
- 局部框架知识加载

这些过程会带来两个问题：

- 无效路径和中间噪音会污染主线程
- 局部领域知识和临时技能会占据主线程上下文

因此，subagent 的本质作用是：

- 把探索过程和执行细节留在子线程
- 只把结果摘要、证据和决策建议带回主线程

## 三、什么情况下应该使用 subagent

### 1. 高噪音探索

适合委派出去的典型工作：

- 代码库探索
- 日志检索
- 报错定位
- 文档搜索
- 网络搜索

判断标准：

- 过程可能有很多错误路径
- 主 agent 不需要知道全部搜索轨迹
- 只需要最终的路径定位、证据和压缩结论

### 2. 强局部领域执行

适合委派的典型场景：

- 某个子模块需要特定 framework 知识
- 某个子任务需要加载特定 skill
- 某个子任务依赖专门的 MCP 或工具链

判断标准：

- 这些知识对整体任务只是局部有效
- 不值得让主 agent 长期背着这一整包上下文

### 3. 独立验证

适合委派的典型场景：

- review
- 风险扫描
- 对抗式检查
- 回归分析
- 独立测试

判断标准：

- 验证线程最好和实现线程隔离
- 避免主线程在“自己证明自己对”的闭环里失真

### 4. 天然可并行的工作

例如：

- 多个风险点独立审查
- 多个文件独立分析
- 多个候选方案独立比较
- 多条 checklist 独立执行

判断标准：

- 子任务之间耦合低
- 输出可以汇总
- 不需要共享大量中间状态

### 5. 当前 Codex 还需要显式 delegation trigger

即使上面几类任务都很适合委派，在当前官方 Codex 行为下，`spawn subagent` 仍然不是自动动作。

这意味着：

- “值得委派” 和 “会自动委派” 不是一回事
- 没有显式授权时，不应该假定系统会自己开 subagent
- 更合理的修正方式是：
  - 提前获得显式 delegation permission
  - 或在任务一开始就用简短问题请求授权
  - 或把授权固化到 task contract 里

## 四、什么情况下不应该使用 subagent

### 1. 子任务太小

- 委派成本高于直接执行成本

### 2. 主线程必须掌握全部细节

- 如果最后还是要把全部中间过程搬回主线程，就没有隔离价值

### 3. 子任务高度串行依赖

- 前一步的中间状态会直接决定下一步动作
- 很难切成稳定边界

### 4. 多个 agent 会改同一批文件

- 并行写入冲突会抵消上下文隔离收益

### 5. 结果无法压缩

- 如果子任务的价值不能以摘要、证据、结构化输出返回，就不适合委派

一句话：

- `不能被压缩返回的工作，不适合委派`

## 五、什么时候值得定义 custom subagent role

只有当下面这些条件大多同时成立时，才值得把一个模式沉淀成 custom role：

### 1. 反复出现

- 不是一次性临时需求
- 会跨任务、跨项目重复出现

### 2. 边界稳定

- 输入稳定
- 输出稳定
- 权限边界稳定
- done definition 稳定

### 3. 需要固定上下文包

例如总是要绑定：

- 同一类 developer instructions
- 同一类 skill
- 同一类 MCP
- 同一类 sandbox / approval 约束

### 4. 主线程不应长期携带这些上下文

- 这些知识是局部有效的
- 如果长期加载在主线程里，会造成上下文浪费

### 5. 输出可以标准化

理想输出通常应能稳定收敛成：

- 结论
- 证据
- 风险
- 改动范围
- 未解决项

## 六、什么类型最值得定义成 custom role

### 1. 治理型工作流

最典型的是：

- 当前性文档同步
- 状态回写
- handoff 收尾
- 结构化验证收口

这类工作不是普通“写文档”，而是：

- 维护上下文分层
- 维护 SSOT
- 维护当前状态一致性

因此这类角色通常最有跨项目价值。

### 2. 强组织特定性的执行者

例如：

- 绑定内部 MCP
- 绑定公司 review 流程
- 绑定内部发布或合规流程

这类东西官方 built-in role 不可能覆盖。

### 3. 输入输出极稳定的流程角色

例如：

- incident triage
- release gate
- compliance scan
- current-doc sync

这类工作天然适合沉淀成固定角色或 skill。

## 七、什么类型通常不值得定义成 custom role

下面这些更适合优先使用 built-in role，而不是额外自定义：

- `explorer`
- `reviewer`
- `docs researcher`

原因不是它们没价值，而是：

- 官方内建能力通常已经足够
- 它们更像通用工作模式
- 真正需要定制的部分，往往更适合放进：
  - `AGENTS.md`
  - review checklist
  - skill
  - MCP 配置
  - 任务级 prompt

## 八、subagent、skill、主 agent 的关系

三者的职责应该分开：

### 1. main agent

- 理解需求
- 决策
- 分解任务
- 审核结果
- 汇总最终输出

### 2. subagent

- 承担局部执行
- 隔离高噪音上下文
- 返回压缩结果

### 3. skill

- 固化方法
- 提供稳定 recipe
- 让同一类工作流可重复执行

所以更好的设计通常不是“每种工作都造一个 custom role”，而是：

- built-in role 负责通用能力
- skill 负责方法沉淀
- 只有少数高价值治理动作再用 custom role

## 九、一个实用的判定框架

在进入具体 checklist 之前，可以先用一个更实用的收益/成本公式做第一层过滤。

### 先算收益

可以把 subagent 的收益理解为：

`subagent 收益 = 上下文隔离收益 + 并行收益 + 专用技能收益 + 权限隔离收益`

这里四项分别对应：

- `上下文隔离收益`
  - 主线程少背多少高噪音探索过程、失败路径和局部知识
- `并行收益`
  - 是否真的能把多个低耦合子任务同时推进
- `专用技能收益`
  - 是否值得把某个 skill、MCP、领域知识包只加载到子线程
- `权限隔离收益`
  - 是否适合把更窄的 sandbox、审批、写入边界限制在子线程中

### 再算成本

同时也要承认 subagent 不是免费能力。

可以把成本理解为：

`subagent 成本 = 委派成本 + 汇总成本 + 协调成本 + 冲突风险`

这里四项分别对应：

- `委派成本`
  - 需要额外写清任务边界、输入和输出契约
- `汇总成本`
  - 主线程要花多少精力审阅和整合返回结果
- `协调成本`
  - 多个子任务之间是否需要等待、回传、重试、串联
- `冲突风险`
  - 是否会出现并行写入、结论不一致、上下文错位、角色漂移

### 实用判断

只有当：

`subagent 收益 > subagent 成本`

时，才值得把这个子任务委派出去。

否则：

- 要么主线程自己做更便宜
- 要么应该先把任务拆得更清楚，再决定是否委派

### 为什么 custom role 的门槛还要更高

把一个模式定义成 `custom role`，不只是一次任务上的委派决策，而是在增加长期维护面。

这意味着你额外承担了：

- 角色命名和边界维护
- 配置文件维护
- 文档同步成本
- 模型、MCP、sandbox 策略随官方变化的老化成本
- 角色和 built-in capability 重叠后的清理成本

所以：

- `值得 spawn subagent`
  不等于
- `值得定义 custom subagent role`

后者的门槛必须再高一层。

先问 5 个问题：

1. 这个子任务是否高噪音，主线程不该保留全过程？
2. 这个子任务是否需要独立的 skill / MCP / 权限包？
3. 这个子任务结果能否压缩成摘要或结构化输出？
4. 这个模式是否会跨项目反复出现？
5. 官方 built-in role 是否已经足够？

### 判断规则

- 如果 `1 + 2 + 3` 大体成立：
  - 值得 `spawn subagent`
- 如果 `1 + 2 + 3 + 4` 成立，且 `5` 不成立：
  - 值得定义 `custom subagent role`
- 如果 `5` 已经成立：
  - 优先不要自定义，先用 built-in role

## 十、对这个仓库方法论的直接含义

如果一个项目强调：

- 分层管理上下文
- 最小主线程
- 任务后文档同步
- durable workflow

那么最值得保留的 custom role，往往不是“更会探索”或“更会 review”的角色，而是：

- `context maintenance agent`
- `currentness sync agent`
- `docs / state / handoff sync agent`

也就是：`docs_syncer` 这类角色。

它的价值不在“写文档”，而在：

- 让任务完成后的上下文治理动作标准化
- 让主线程不必长期携带整套当前性同步逻辑
- 让分层上下文系统在多项目里保持一致

同时还要补一层现实约束：

- 当前 Codex 产品不会自动 spawn subagent
- 所以仓库方法论不能只停留在“什么时候值得委派”
- 还需要一套低摩擦的显式 delegation contract
- 否则主 agent 即使判断“该委派”，也可能因为没有授权而保持单线程执行

## 最终判断

subagent 最重要的价值是 `隔离局部高熵上下文`。

custom subagent role 最重要的价值是 `沉淀跨项目可复用的上下文治理模式`。

因此：

- 在显式授权已经存在时，`spawn subagent` 应该相对常见
- 没有显式授权时，更好的默认动作是主动提出 delegation 建议，而不是假定会自动委派
- `define custom role` 应该相对稀少

真正值得长期保留成 custom role 的，通常不是“能力名字”，而是“治理动作”。
